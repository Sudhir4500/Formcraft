🏗️
FormCraft — production architecture
Django REST backend · Next.js BFF frontend · SEO-optimized · standardized API responses · Stripe billing
🐍
Django REST backend
Pure API server. Auth, forms, responses, billing. Never renders HTML. All output goes through ApiResponse helper.
🔁
Next.js as BFF
Next.js Route Handlers act as a Backend for Frontend — proxies Django, attaches tokens from HttpOnly cookies, shapes data for UI needs.
🔍
SEO-first routing
Every public page (landing, pricing, /f/[slug]) uses SSR or SSG with full meta tags, OG tags, JSON-LD, and sitemap.
🛡️
Standardized responses
Every API response — success or error — always returns the same 4-field shape. Frontend never guesses the structure.
💡
Why BFF? The Django API is a pure data service — it doesn't know about cookies, Next.js sessions, or per-page data shapes. The BFF layer (Next.js Route Handlers) sits between the browser and Django: it reads the HttpOnly cookie, attaches the JWT to outgoing Django requests, and can combine multiple Django calls into one browser request. This is the same pattern used by Vercel, Linear, and most modern SaaS products.

🐍 core/responses.py 
from rest_framework.response import Response
from rest_framework import status as http_status

class ApiResponse:
    """
    Single source of truth for all API responses.
    Usage:  return ApiResponse.success(data, "Created")
            return ApiResponse.validation_error(serializer.errors)
    """

    @staticmethod
    def success(data=None, message="", status_code=http_status.HTTP_200_OK):
        return Response({
            "success": True,
            "message": message,
            "data":    data,
            "errors":  None,
        }, status=status_code)

    @staticmethod
    def created(data=None, message="Created successfully"):
        return ApiResponse.success(data, message, http_status.HTTP_201_CREATED)

    @staticmethod
    def error(message="Something went wrong", errors=None,
              status_code=http_status.HTTP_400_BAD_REQUEST):
        return Response({
            "success": False,
            "message": message,
            "data":    None,
            "errors":  errors,
        }, status=status_code)

    @staticmethod
    def validation_error(serializer_errors):
        # Flatten DRF nested error lists → { field: "first error message" }
        flat = {}
        for field, msgs in serializer_errors.items():
            flat[field] = msgs[0] if isinstance(msgs, list) else str(msgs)
        return ApiResponse.error("Validation failed", errors=flat)

    @staticmethod
    def not_found(message="Not found"):
        return ApiResponse.error(message, status_code=http_status.HTTP_404_NOT_FOUND)

    @staticmethod
    def forbidden(message="Permission denied"):
        return ApiResponse.error(message, status_code=http_status.HTTP_403_FORBIDDEN)

    @staticmethod
    def unauthorized(message="Authentication required"):
        return ApiResponse.error(message, status_code=http_status.HTTP_401_UNAUTHORIZED)
🐍 core/exceptions.py 
from rest_framework.views import exception_handler

def custom_exception_handler(exc, context):
    # Call DRF's default handler first
    response = exception_handler(exc, context)

    if response is not None:
        # Rewrite into our standard shape
        original = response.data
        response.data = {
            "success": False,
            "message": _extract_message(original),
            "data":    None,
            "errors":  original,
        }
    return response

def _extract_message(data):
    if isinstance(data, dict):
        return str(data.get("detail", "An error occurred"))
    return "An error occurred"

BFF — Django side
Django stays clean: pure data API, no knowledge of the frontend. The BFF lives entirely in Next.js.
⚠️
What Django does NOT know about: cookies, sessions, frontend routing, or per-page data shapes. Django only knows: "I received a valid JWT Bearer token, here is your data." The BFF layer handles everything else.

🔀
BFF layer — Next.js Route Handlers
The browser never calls Django. It calls /api/* on Next.js, which reads the cookie, attaches the token, and proxies to Django.
🔐
Security reason for BFF: The JWT token lives in an HttpOnly cookie — JavaScript cannot read it. The BFF Route Handler runs on the server, reads the cookie from the request headers, and adds it as a Bearer header to the outgoing Django call. This means the token is never exposed to client-side JS at all.
⚛️ app/api/_lib/django.ts — Django proxy helper 
import { cookies } from 'next/headers'
import type { ApiResponse } from '@/types/api'

const DJANGO_URL = process.env.DJANGO_INTERNAL_URL // server-only, no NEXT_PUBLIC_

export async function djangoGet<T>(path: string): Promise<ApiResponse<T>> {
  const token = cookies().get('access_token')?.value
  const res = await fetch(`${DJANGO_URL}${path}`, {
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
    },
    cache: 'no-store', // always fresh for auth'd data
  })
  return res.json()
}

export async function djangoPost<T>(path: string, body: unknown): Promise<ApiResponse<T>> {
  const token = cookies().get('access_token')?.value
  const res = await fetch(`${DJANGO_URL}${path}`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
    },
    body: JSON.stringify(body),
  })
  return res.json()
}
⚛️ app/api/forms/route.ts — BFF Route Handler 
import { NextRequest, NextResponse } from 'next/server'
import { djangoGet, djangoPost } from '@/app/api/_lib/django'
import type { Form } from '@/types/forms'

// GET /api/forms  → proxies to Django GET /api/forms/
export async function GET() {
  const data = await djangoGet<{ results: Form[] }>('/api/forms/')
  return NextResponse.json(data)
}

// POST /api/forms  → proxies to Django POST /api/forms/
export async function POST(req: NextRequest) {
  const body = await req.json()
  const data = await djangoPost<Form>('/api/forms/', body)
  return NextResponse.json(data, {
    status: data.success ? 201 : 400,
  })
}

// The browser calls /api/forms — never Django directly.
// Token is never exposed to client-side JavaScript.
⚛️ app/api/auth/login/route.ts — login + cookie set 
import { NextRequest, NextResponse } from 'next/server'
import { djangoPost } from '@/app/api/_lib/django'

export async function POST(req: NextRequest) {
  const body = await req.json()
  const data = await djangoPost<{ access: string; refresh: string; user: object }>(
    '/api/auth/login/', body
  )

  if (!data.success || !data.data) {
    return NextResponse.json(data, { status: 401 })
  }

  const res = NextResponse.json({ success: true, message: data.message, data: { user: data.data.user }, errors: null })

  // Set HttpOnly cookie — JS can never read this
  res.cookies.set('access_token', data.data.access, {
    httpOnly:  true,
    secure:    process.env.NODE_ENV === 'production',
    sameSite:  'lax',
    maxAge:    60 * 15, // 15 min
    path:      '/',
  })
  res.cookies.set('refresh_token', data.data.refresh, {
    httpOnly: true, secure: process.env.NODE_ENV === 'production',
    sameSite: 'lax', maxAge: 60 * 60 * 24 * 7, path: '/',
  })

  return res
}
